# 代码格式概念说明

用于统一 `Driver IC 初始化代码` 区域中左右两个文本框、按钮文案、tooltip 与提示语义，避免后续继续混淆。

## 四个概念

### 1. 原始代码

指用户直接粘贴进来的原始内容，可能包含：

- 注释
- 多余分隔符
- 不统一空格
- `0x` 前缀
- `REGWxx` 风格
- `delay` / `delayms`
- 其他历史写法

这类内容**不保证可以直接下发**，也**不保证可以直接用于 OLED config bin**。

### 2. 清洗后代码

指对原始代码做基础清洗后的结果，例如：

- 去注释
- 统一大小写
- 统一空白和分隔符
- 去掉无意义符号

清洗后代码只是更规整，**不等于已经满足目标格式**。

### 3. 标准代码

指经过归一化后、适合继续转换和编辑的统一表示。

特点：

- 可以接受 `delay / delayms`
- 可以接受 `REGW05 / REGW39 / REGW29 / REGW0A`
- 会被统一规整成标准命令表示
- 右侧文本框应主要承载这种“便于编辑和继续转换”的表示

右侧文本框及其相关操作，主要围绕“能否转换为标准代码”展开。

### 4. 格式化代码

指左侧文本框中的代码，也就是：

- 用于 `vismpwr` 命令检查/下发
- 用于生成 `OLED config bin`
- 面向最终初始化序列的格式化结果

左侧文本框不是原始草稿区，而是更接近最终使用形态的代码区。

## 左右两个文本框的职责

### 左侧文本框

用途：

- 保存格式化代码
- 用于 OLED config bin 生成
- 用于 `vismpwr` / 初始化代码下发相关检查

#### 左侧检查按钮语义

按钮文案：`vismpwr检查`

tooltip：`检查左侧格式化代码是否符合 vismpwr 可下发格式`

左侧检查要确保：

1. 左文本框里的代码格式符合 `vismpwr` 写命令要求
2. 每行字段合法
3. `DT` 目前仅允许 `05 / 29 / 39 / 0A`
4. 长度字段和后续数据数量一致
5. 基础 DT 约束成立，例如：
   - `DT=05` 时 `LEN` 必须为 `01`
   - `DT=29` 时通常 `LEN` 至少为 `02`；若 `LEN=01`，给 warning，提醒人工确认，不直接报错
   - `DT=39` 时通常 `LEN` 至少为 `02`；若 `LEN=01`，给 warning，提醒人工确认，不直接报错
   - `DT=0A` 时 `LEN` 至少为 `01`
6. 左侧保留的是最终格式化代码，而不是带 `delay` 关键字的草稿表达
7. 当前检查本身不强制要求存在 `DT=0A`；但在 **DSC 已启用** 且执行 **OLED 配置下载** 时，会额外强制校验 `DT=0A` 数据行是否存在

也就是说：

> 左侧 vismpwr检查 = 检查“格式化代码 / 可下发代码”是否合规

#### 左 -> 右 按钮语义

tooltip：`将左侧格式化代码还原为右侧标准代码（05/29/0A -> REGWxx，delay 单独展开）`

含义：

- 左侧是格式化代码区
- 右侧是标准代码 / 草稿区
- 因此左推右不是“原样复制”，而是：
  - 先按“格式化代码 -> 标准代码”语义还原
  - 再写入右侧

当前还原规则：

- `39 00 LEN DATA...` -> 右侧裸数据行 `DATA...`
- `05 00 LEN DATA...` -> `REGW05 DATA...`
- `29 00 LEN DATA...` -> `REGW29 DATA...`
- `0A 00 LEN DATA...` -> `REGW0A DATA...`
- 若第二字段 delay 不为 `00`，则在当前命令后追加一行 `delay N`

示例：

```text
05 00 01 29
```

还原为：

```text
REGW05 29
```

```text
0A 00 05 12 34 56 77 88
```

还原为：

```text
REGW0A 12 34 56 77 88
```

```text
05 10 01 29
```

还原为：

```text
REGW05 29
DELAY 16
```

成功提示建议统一为：

> 已将左侧格式化代码还原为标准代码并填充到右侧：共 N 行

### 右侧文本框

用途：

- 作为独立草稿区
- 接收原始代码 / 清洗后代码 / 待转换代码
- 允许用户编辑、整理、检查、转换
- 保存标准代码更合适，而不是保存最终下发格式

#### 右侧检查按钮语义

按钮文案：`可转换性检查`

tooltip：`检查右侧原始代码是否可以清理并转换为标准代码`

右侧检查不是检查它是否已经满足 `vismpwr` 下发要求，
而是检查：

- 当前右侧内容是否能被识别
- 是否能被清洗并归一化为标准代码
- 是否存在无法转换的字段或非法内容
- 若检测到“疑似格式化代码样式输入”，必须是该行**严格符合左侧格式化代码条件**（`DT / DELAY / LEN / DATA...` 完整成立，且 `LEN` 与 payload 数量、DT 约束都成立），此时不直接判错，只给出警告，提醒用户人工重点检查

也就是说：

> 右侧可转换性检查 = 检查“输入源码”能否安全转换为标准代码

成功提示建议统一为：

> 可转换性检查通过：可转换为 N 行标准代码

若检测到疑似格式化代码样式输入（且该行严格符合左侧格式化代码条件），则提示建议统一为：

> 可转换性检查通过，但检测到 N 行疑似格式化代码样式输入，请人工重点检查

失败提示建议统一为：

> 可转换性检查未通过：共 N 处问题

#### 右侧“标准化转换”按钮语义

按钮文案：`标准化转换`

tooltip：`将右侧原始代码清理并转换为标准代码后回填右侧`

含义：

- 这是“右侧原地转换”
- 输入是原始代码 / 草稿代码
- 输出是标准代码
- 转换结果仍写回右侧文本框

成功提示建议统一为：

> 已将右侧原始代码清理并转换为标准代码：共 N 行

若本来已经是标准代码，则提示：

> 右侧内容已是标准代码：共 N 行

#### 右 -> 左 按钮语义

tooltip：`将右侧内容清理并转换为格式化代码后填充到左侧`

含义：

- 右侧推到左侧时，不能直接原样复制
- 必须先把右侧原始代码 / 标准代码整理成左侧使用的格式化代码
- 再更新左侧文本框

成功提示建议统一为：

> 已将右侧内容清理并转换为格式化代码后填充到左侧：共 N 行

## 关键约束

### 左侧和右侧不是同一个状态

右侧文本框应被视为：

- 独立草稿区
- 不应在切页、自动恢复配置时被左侧内容隐式覆盖

只有明确用户动作才允许把左侧内容写入右侧，例如：

- 左 -> 右 按钮
- 用户主动加载配置且明确要求同步右侧

### 左侧不保留 delay 文本

虽然原始代码和标准代码阶段可以存在 `delay / delayms` 语义，
但左侧作为格式化代码区，最终必须是：

- 符合 `vismpwr` 使用要求
- delay 语义应折叠进上一条命令的第二字段
- 不再显式保留 `delay` 文本

## vismpwr.c 实际支持格式整理

来源：`E:\Resource\8Big8K\8K_software\linux_rk3588\vismpwr.c`

### 1. vismpwr 写命令格式

`vismpwr` 写模式使用：

```bash
vismpwr [-0|-1] data...
```

源码示例：

```bash
vismpwr 29 00 03 51 12 34
```

从 `parse_args()` 可以确认，写模式实际要求的参数结构为：

```text
DT DELAY LEN DATA...
```

对应含义：

- 第 1 字段：`DT`，数据类型（1 字节）
- 第 2 字段：`DELAY`，延时字节（1 字节）
- 第 3 字段：`LEN`，后续有效负载长度（1 字节）
- 第 4 字段起：`DATA...`，数据内容，共 `LEN` 个字节

### 2. vismpwr 写模式的关键校验

源码中的关键逻辑：

- `payload_len = parse_hex_byte(argv[data_start+2])`
- 然后循环读取 `payload_len + 3` 个字段
- 最终下发长度为 `1 + 1 + 1 + payload_len`

因此可以确认：

1. 左侧每行至少需要 4 个字段
2. 第 3 字段 `LEN` 必须与后续数据个数完全一致
3. 每个字段都必须是 1 个或 2 个十六进制字符，最终会被解析成 1 字节
4. 支持单字符十六进制写法，例如 `1` 会被解析为 `0x01`
5. 当前 UI 规范仍统一输出两位大写十六进制，避免歧义

### 3. vismpwr 支持的设备选择参数

源码支持：

- `-0`：`/dev/mipi_dsi0`
- `-1`：`/dev/mipi_dsi1`

UI 当前主要关注命令主体文本，不在左侧文本框内混入设备选择参数。

### 4. vismpwr 读命令格式

源码 usage：

```bash
vismpwr -r [-0|-1] read_len read_reg
```

示例：

```bash
vismpwr -r 02 51
```

其中：

- `read_len`：读取长度
- `read_reg`：寄存器地址

这属于读路径，不属于当前左侧“格式化代码”写入序列的文本格式范围。

### 5. vismpwr.c 没有直接支持的语法

`vismpwr.c` 本身并不直接解析这些关键字：

- `delay`
- `delayms`
- `REGW05`
- `REGW29`
- `REGW39`
- `REGW0A`

这些都是 **UI 层为了兼容历史写法和提升可编辑性** 定义的上层语义。
UI 需要先把这些语义转换为 `vismpwr` 实际接受的字节行格式，再用于左侧检查、下发和 OLED config 生成。

## 当前实现约定

### 标准代码可接受的上层写法

当前前端归一化逻辑支持：

- `delay N`
- `delayms N`
- `REGW05 ...`
- `REGW29 ...`
- `REGW39 ...`
- `REGW0A ...`
- 裸十六进制数据行（默认补成 `39 00 LEN ...`）
- 已经是标准格式 / 格式化格式的 `05|29|39|0A 00 LEN ...`

### delay 的折叠规则

当前实现中：

- `delay` / `delayms` 不能出现在第一行
- 它不会单独保留成一行
- 会把延时值写入上一条命令的第二字段

例如：

```text
REGW05 11
DELAY 78
```

会被归一化为：

```text
05 4E 01 11
```

其中 `4E` 为十进制 78 的十六进制表示。

## 统一命名

- 左侧按钮文案：`vismpwr检查`
- 右侧按钮文案：`可转换性检查`
- 右侧转换按钮文案：`标准化转换`
- 顶部下发按钮文案：`代码下发`
  - tooltip：`将右侧原始代码或标准代码转换为 vismpwr 命令后下发`
  - 可见说明：`下发时将自动转换为 vismpwr 命令`
- `OLED 配置下载` 的前置规则：
  - 点击后先执行左侧格式化代码校验
  - warning 不拦截下载，只有 error 才中止下载
  - 若 `DSC Enable = true`，则左侧初始化代码中必须存在至少一行 `DT=0A`
  - 若缺少 `DT=0A`，必须明确提示用户后中止下载
- `导出 OLED 配置(JSON)` 的行为：
  - 复用同一套左侧前置校验
  - 导出 `TimingBinRequest` 对应 JSON
  - `vis-timing.bin` 仍按原逻辑默认保存在程序目录，不再额外复制 sidecar bin

## 实现命名约定

为避免 UI 语义与代码命名再次跑偏，动作函数按职责命名：

- `handleVismpwrCheckAction`
- `handleRightConvertibilityCheckAction`
- `handleNormalizeToStandardAction`
- `handleFormattedToStandardAction`
- `handleStandardToFormattedAction`
- `handleSendRightEditorAction`

其中：

- `SendRightEditor` 明确表示“从右侧内容发出”
- `FormattedToStandard` 表示“左侧格式化代码 -> 右侧标准代码”
- `StandardToFormatted` 表示“右侧原始/标准代码 -> 左侧格式化代码”

若后续需要，再把这套定义同步进 README 的开发说明部分。
